# To Issues — план → tracer-bullet слайсы

Разбивает план на **вертикальные слайсы** (tracer bullets): каждый — тонкий, но СКВОЗНОЙ
путь через все слои (схема → API → UI → тесты), демонстрируемый сам по себе. НЕ
горизонтальные слои («сначала вся схема, потом весь API»).

## Процесс

1. **Контекст**: работай с тем, что уже в разговоре; передали issue/файл — прочитай
   целиком. Если код не смотрел — посмотри: возможен префакторинг («сделай изменение
   лёгким, потом сделай лёгкое изменение»).
2. **Черновик слайсов**: для каждого — название, blocked by (какие слайсы раньше), что
   демонстрирует по завершении.
3. **Квиз пользователю**: покажи нумерованным списком. Спроси про гранулярность
   (крупно/мелко?), зависимости, что слить/разбить. Итерируй до одобрения.
4. **Публикация** (в порядке зависимостей, blockers первыми):
   - целевой репо — тот, про который план (сомневаешься — правила `/route`);
   - **инварианты хаба**: родительский epic (найди/создай, свяжи через Sub-issues),
     лейбл `dept:<роль репо из HQ.md>`, добавить на Project-доску (`project_number`
     из Hub Config);
   - W-лейбл — только если уже существует (создаёт `/weekly`);
   - тело: цель слайса, «Blocked by: <repo>#N», критерий демо, ссылки на план/дамп.
5. Ответ: список `<repo>#N «title»` в порядке работы.

## Red flags

| Собираешься… | Вместо этого… |
|---|---|
| Слайс «сделать всю БД-схему» | Вертикальный кусок: одна фича сквозь все слои |
| Публиковать без квиза | Шаг 3: гранулярность подтверждает человек |
| Issues без epic/dept-лейбла/доски | Инварианты шага 4 |
| Ссылаться «слайс 3» | Только `<repo>#N «title»` |
