# Route — задача в правильное место

Целевой репо определяют **ключевые слова задачи**, не текущая папка. Никогда не дефолти
в текущий репо.

## Шаги

1. **Конфиг**: таблица `HQ.md` штаба — колонки «репо», «роль», «ключевые слова (routing)».
2. **Матчинг**: слова задачи против колонки ключевых слов. Несколько совпало — самый
   специфичный. Ничего — покажи реестр и спроси; предложи дополнить ключевые слова
   строки в HQ.md (коммит штаба).
3. **Дедуп**: `gh issue list -R <target> -s open` + поиск по доске. Похожее есть →
   предложи обновить его, не плодить дубль.
4. **Инварианты issue**:
   - лейбл роли `dept:<role>` (роль репо из HQ.md; лейбла нет в репо — создай);
   - **ровно один родительский epic**: найди открытый epic репо/направления
     (`gh issue list -l epic`); нет — предложи создать; свяжи через sub-issue
     (`gh api` Sub-issues) или строкой `Parent: <repo>#N` в теле;
   - W-лейбл текущей недели — только если уже существует в репо (создаёт /weekly);
   - заголовок: `{объект} — {действие} ({контекст})`; домен — лейблами, не в заголовке.
5. **Создание**: `gh issue create -R <target> -t "..." -l "..." --body "..."` →
   добавь на Project-доску (`gh project item-add <number> --owner <owner> --url <issue-url>`).
6. Ответь ссылкой: `<repo>#N «title»`.

## Red flags

| Собираешься… | Вместо этого… |
|---|---|
| Issue в текущем репо «потому что я тут» | Матчинг по словам; нет матча — спросить |
| Создать без проверки дублей | Шаг 3 |
| Issue без epic и dept-лейбла | Шаг 4 обязателен |
| Создать W-лейбл руками | Пропустить — его даст /weekly |
