# Handoff — передача сессии через GitHub

Память системы — GitHub, поэтому handoff живёт **комментарием в issue**, а не во
временном файле: следующая сессия на любой машине найдёт его.

## Процесс

1. **Куда писать**: активный issue этой работы (по которому шла сессия). Нет issue —
   создай по правилам `/route` (работа без issue = невидимая работа).
2. **Сначала артефакты**: незакоммиченное — закоммить (в ветку, если main опасен);
   незапушенное — запушь. Handoff, ссылающийся на локальное состояние, бесполезен.
3. **Комментарий-handoff** (`## Handoff → следующая сессия`):
   - **Состояние**: что сделано / что осталось (чек-лист);
   - **Где я остановился**: файл/строка/команда, последняя мысль;
   - **Грабли**: что уже пробовал и не сработало (сэкономит следующему часы);
   - **Следующий шаг**: первая конкретная команда/действие;
   - **Ссылки**: коммиты `<repo>@<sha>`, PR, диффы — НЕ дублируй их содержимое;
   - **Рекомендуемые скиллы** для продолжения;
   - аргумент пользователя («чем займётся следующая сессия») — учти фокус.
4. **Редактируй чувствительное**: никаких токенов/паролей/личных данных в issue.
5. Статус на доске оставь In Progress; скажи пользователю: «продолжить — открой
   <repo>, скажи агенту прочитать <issue-url>».

## Red flags

| Собираешься… | Вместо этого… |
|---|---|
| Handoff в файл/tmp | Комментарий в issue — переживает машину |
| Пересказывать содержимое диффов | Ссылки на коммиты/PR |
| Ссылаться на незапушенный коммит | Шаг 2: сначала push |
| Вставить API-ключ «для удобства» | Имя переменной + где лежит значение |
