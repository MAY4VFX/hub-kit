# Implement — слайс до коммита

Тонкая обвязка цикла реализации. Вход: номер issue (`<repo>#N`) или «следующий готовый».

## Процесс

1. **Возьми задачу**: без номера — `gh issue list -l ready-for-agent` целевого репо,
   старейший неблокированный. Прочитай issue целиком: бриф, Blocked by (не закрыты —
   STOP, возьми другой), definition of done.
2. **Свяжи работу**: назначь issue на себя-пользователя, статус на доске → In Progress.
3. **Реализация**:
   - тесты вперёд там, где есть договорённый шов (definition of done подсказывает);
   - типы/линт/один тест-файл — регулярно; полный прогон — один раз в конце;
   - границы из брифа не пересекать; вылез за рамки — останови, запиши в issue, спроси.
4. **Ревью перед коммитом**: запусти доступные инструменты ревью (в Claude Code:
   `/code-review`, `/verify`; иначе — самопроверка диффа против definition of done).
5. **Коммит(ы)**: conventional commits, атомарно, в текущую ветку. Push — по правилам
   репо (авто-деплой с main → спроси или веткой+PR).
6. **Work-record** в issue: что сделано + `<repo>@<sha>` + как проверял. Definition of
   done выполнен → закрой issue, статус на доске → Done.

## Red flags

| Собираешься… | Вместо этого… |
|---|---|
| Начать без чтения брифа целиком | Бриф = контракт; нет брифа → /triage |
| Работать по заблокированному слайсу | Возьми неблокированный |
| Закрыть issue без work-record | Комментарий с коммитами обязателен |
| Полный тест-прогон после каждой строчки | Точечно в процессе, полный — в конце |
