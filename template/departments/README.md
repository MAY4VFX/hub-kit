# Стандарт отдела

Регламент: как устроен отдел (роль), как создать новый, какие файлы в нём возможны.
Отделы создают ТОЛЬКО скиллы `/hub-init` (при онбординге) и `/dept --new <роль>` —
оба следуют этому стандарту. Агент не выдумывает свою структуру.

## Структура

```
departments/<slug>/          # slug: латиница, kebab-case
├── role.md                  # ОБЯЗАТЕЛЬНЫЙ. Единственный файл-слой: на него смотрит
│                            # симлинк .dept.md проектов, он попадает в контекст агента
├── <материалы>.md           # опциональные вопрос-файлы, по белому списку ниже
└── docs/                    # знания роли: ресёрчи, гайды, post-mortem, справки
```

## Материалы отдела — что они дают агенту

Каждый файл отвечает на ОДИН конкретный вопрос агента, работающего в этой роли.
Пишутся ДЛЯ агента: императив, готовые команды, никакой прозы. Создавать по мере
надобности, не заранее.

| Файл | Вопрос агента | Что внутри | Кто обновляет |
|---|---|---|---|
| `map.md` | «Что у нас есть и где?» | карта ресурсов роли: хосты, сервисы, эндпоинты, аккаунты, где лежат креды (имена env-переменных) | профильный скилл-скан (напр. `/infra-scan`) или явная задача |
| `recipes.md` | «Как сделать X?» | проверенные рецепты с готовыми командами: задеплоить, посмотреть логи, перезапустить, прогнать миграции | явная задача или `/learn`, когда урок = процедура |
| `gotchas.md` | «Обо что тут разбиваются?» | грабли: что нельзя делать, что от этого ломается, как чинить; каждая — из реального инцидента | `/learn` (главный поставщик), post-mortem инцидентов |
| `skills.md` | «Каким скиллом это делать?» | карта «ситуация → скилл» роли: когда какой скилл запускать и зачем (например: инцидент → /diagnosing-bugs, деплой → /dokploy) | явная задача; пополняется, когда роль получает новый скилл |
| `stack.md` | «Чем и почему мы работаем?» | дефолтные инструменты/библиотеки/сервисы роли + одной строкой почему выбраны (чтобы агент не предлагал альтернативы заново) | явная задача |

Нужен файл вне списка — сначала добавь его сюда (в стандарт) с вопросом и владельцем,
потом создавай.

## `docs/` — знания роли

Свободная папка глубоких знаний: результаты `/research`, гайды, post-mortem инцидентов,
справки. В отличие от вопрос-файлов (короткие, курируемые) — здесь полные документы.

- Имя: `YYYY-MM-DD-<тема-kebab>.md` — дата даёт сортировку и понимание свежести.
- Первые 3 строки каждого дока — TL;DR: агент решает, читать ли дальше, не читая всё.
- `/research` по теме роли сохраняет отчёт СЮДА (по теме конкретного проекта — в docs/ репо проекта).
- Вопрос-файлы ссылаются на доки: рецепт в recipes.md может вести на полный гайд в docs/.
- Папку не создавать заранее — она появляется с первым документом.

## Одноимённый скилл роли

У каждой роли есть тонкий скилл-обёртка `skills/<slug>/SKILL.md` (из
`_dept-hat.md.tmpl`): команда `/<slug>` надевает эту роль на сессию (эфемерная шляпа =
`/dept <slug> --here`). Создаётся автоматически вместе с ролью (`/hub-init`, `/dept --new`).
Новый скилл появляется после `plugin update` + новой сессии.

## Правила

1. **В контекст автоматически попадает только `role.md`** — держи его коротким
   (< 40 строк). Всё тяжёлое — в соседние файлы, из role.md — ссылка одной строкой
   («карта ресурсов — map.md»); агент читает их по ссылке, когда нужно.
2. **role.md не перечисляет проекты** — проекты роли смотри в HQ.md (иначе устареет).
3. **Секретов в папке отдела нет никогда** — только имена переменных.
4. У каждого материала есть владелец (скилл или «явная задача») — файлы без владельца гниют.

## Секции role.md (шаблон `_role.md.tmpl`)

```markdown
# Роль: <имя>            ← identity: «Ты — агент роли X. Штаб: <hub>»
## Зона ответственности   ← 1–3 строки + указатель на HQ.md
## Правила роли           ← пополняется /learn; императив, с «почему»
## Задачи                 ← лейбл dept:<slug>, где смотреть открытые
## Материалы отдела       ← ссылки на файлы папки (если есть)
## Рекомендуемые скиллы   ← чем эта роль обычно работает
```

## Чек-лист создания нового отдела (`/dept --new` делает сам)

1. Спросить у владельца: имя, slug, домен одной строкой.
2. `departments/<slug>/role.md` из `_role.md.tmpl` (примеры — `examples/` в hub-kit).
3. Лейбл `dept:<slug>` в hub-репо.
4. Коммит + push штаба. Материалы — НЕ создавать заранее (пустые файлы = мусор).
